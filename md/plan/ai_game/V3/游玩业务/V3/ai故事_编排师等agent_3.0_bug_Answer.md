# 编排师等agent_3.0 bug 回答

## 1. 关于“编排师不作为，编排机制全废”

这个判断基本成立。

当前 `3.0` 的问题不是“完全没有编排代码”，而是规则层介入过深，导致很多本该由编排师判断的轮次，被 `RuleOrchestrator` 直接短路掉了。

也就是说：

- 不是没有编排器
- 而是编排器出场频率被压得过低
- 结果表现出来就像“没有编排”

## 2. 关于“phase 已经强约束为 某男子 -> 萧炎 -> 旁白”

字面上不是写死成这一条固定链，但效果上已经接近。

现在的顺序不是由模型每轮动态判断，而是被这些规则大量接管：

- `expected_role`
- `phase_named_role`
- `phase_wildcard_role`
- `narrator_bridge`
- `single_allowed_speaker`

再加上 `resolveNextRuleRole()` 默认会把很多轮次导向“旁白 / 另一个 NPC”，所以最后呈现出来就像“半硬编码顺序推进”。

你的感觉是对的。

## 3. 关于“编排师和记忆管理师完全罢工”

不完全准确，但业务体感上接近成立。

当前状态是：

- 编排师还在
- 记忆管理师也还在
- 但规则命中时，编排师直接不跑
- `template / fast / rule` 轮次里，记忆刷新又经常被压掉

所以从日志和表现上，会很像：

- 只剩角色发言器在工作
- 编排师和记忆管理师存在感极低

问题不是“模块被删了”，而是“触发频率被压得太低”。

## 4. 关于“规则优先导致编排机制崩溃”

这个结论成立。

真正的问题不是“规则优先”这个原则本身，而是当前规则层承担了太多本该由 AI 编排师处理的判断。

规则本来应该只吃：

- 明确唯一角色
- 明确用户回合
- 明确固定过场

但现在已经扩到了：

- phase 文本点名
- 阶段桥接
- 万能角色兜底
- 旁白补句
- 单角色阶段连续承接

这就导致 AI 编排被大量短路，最后表现成编排机制塌掉。

## 5. 关于“记忆更新从每轮刷新改成按事件刷新，到底靠什么判断”

答案是：**当前是混合判断，但整体偏硬编码，而且这套策略还不够成熟。**

当前来源有两类：

### 5.1 AI 判断

编排师/发言链仍然可以返回：

- `memoryHints`
- `triggerMemoryAgent`

也就是说，AI 仍然可以显式判断“这一轮值得更新记忆”。

### 5.2 规则与路由层判断

当前代码里，很多轮次会因为属于：

- `rule`
- `template`
- `fast`

而默认不触发记忆刷新，或者显著降低触发概率。

这部分本质上就是规则侧的硬判断。

所以你问“靠硬编码还是 AI 分析”，当前准确答案是：

- **有 AI 分析**
- **但最终整体行为已经明显偏向规则/硬编码控制**

## 当前真正的问题总结

`3.0` 当前不是“只优化了性能”，而是已经影响到了业务正确性。

核心问题有三个：

1. `RuleOrchestrator` 命中面过大，抢走了太多 AI 编排轮次
2. 发言分级把 `template / fast / premium` 做出来了，但没有把业务边界收清楚
3. 记忆刷新从“每轮”改成“按事件”后，缺少一套清晰、稳定、可解释的判定标准

## 建议的纠偏方向

### P0：先恢复业务正确性

- 缩小 `RuleOrchestrator` 命中面
- 恢复 AI 编排主导
- 只保留真正确定性的规则直出

### P1：重新定义“按事件刷新”

明确只有以下事件才允许跳过或触发记忆刷新：

- 新人物关系变化
- 新任务
- 新道具/状态变化
- 章节阶段切换
- 用户明确表态或关键选择

不要再让 `rule / template / fast` 的副作用去决定记忆是否刷新。

### P2：重新定义发言分级的边界

- `template` 只给纯过场、纯氛围、纯功能句
- `fast` 只给低风险普通承接句
- 关键角色冲突、立场、转折、关系变化必须走 AI 编排 + 较完整 speaker

## 结论

这份 bug 文档的主判断是对的。

当前 `3.0` 最大的问题不是“性能优化失败”，而是：

**为了提速，把过多业务判断下沉到了规则层，导致编排、记忆、发言三者的边界被打乱了。**

下一步不应该继续给 `3.0` 叠补丁，而应该做一轮纠偏：

- 收回规则层权限
- 恢复编排师主导
- 把记忆刷新规则写清楚
