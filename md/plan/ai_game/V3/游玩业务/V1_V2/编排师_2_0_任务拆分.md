## 目标
把编排师 2.0 方案拆成可以逐步实施的任务，优先解决：
- 剧情脱离章节
- 用户节点不稳定
- 结束条件不可靠
- 最近对话权重过高
- 编排师职责过大

## 当前进度
- `任务 1` 已完成基础版：`chapterProgress` 已进入会话状态归一化
- `任务 2` 已完成一阶段：保存章节时会持久化最小 `runtimeOutline`，旧章节启动时会自动回填
- `任务 3` 已完成基础版：已新增 `ChapterProgressEngine`，并接入 `startSession / SessionService`
- `任务 3` 已推进一阶段：`runtimeOutline` 已支持最小 `phases`，`chapterProgress.phaseId/phaseIndex` 已和章节阶段对齐
- `任务 5` 已推进一阶段：自动推进已接入最小阶段推进逻辑，进入用户阶段时会主动停下而不只依赖 `awaitUser`
- `任务 5` 已继续推进：阶段推进已改为基于 `completedEvents` 的最小事件化模式，非用户阶段会先打完成标记再选择下一阶段
- `任务 5` 已继续推进：触发器命中、任务进度、状态变更信号开始回灌 `completedEvents`，用于驱动固定事件和当前阶段完成判定
- `任务 5` 已继续推进：`runtimeOutline.phases` 已补最小 `advanceSignals / relatedFixedEventIds`，自动推进不再对所有非用户阶段做盲目前进
- `任务 5` 已继续推进：`runtimeOutline.phases` 已补最小 `requiredEventIds / completionEventIds`，阶段切换开始具备前置依赖和完成事件语义
- `任务 5` 已继续推进：章节正文里的 `@允许角色 / @阶段前置 / @阶段完成 / @阶段信号 / @关联结果` 已可直接编译到 `runtimeOutline.phases`
- `任务 5` 已完成当前阶段：`runtimeOutline.phases` 已补 `nextPhaseIds / defaultNextPhaseId`，阶段切换不再只按线性顺序推进
- `任务 6` 已推进一阶段：编排师候选角色和发言器当前角色都开始受 `phase.allowedSpeakers` 约束
- `任务 6` 已继续推进：编排模型异常和发言模型异常时也会走 phase 过滤后的 fallback，不再绕开当前阶段角色白名单
- `任务 6` 已继续推进：`nextRole / nextRoleType` 也开始按当前阶段允许角色收口，不再把下一棒传给阶段外角色
- `任务 7` 已开始基础版：已新增 `ChapterOutcomeEngine`，先把 `completionCondition / runtimeOutline` 转成硬判定入口
- `任务 7` 已继续推进：正式游玩链已关闭模型直控 `chapterOutcome / nextChapterId`，改为规则引擎优先、模型只保留调试提示能力
- `任务 7` 已完成当前阶段：`debugStep / orchestration` 调试链也已接入 `ChapterOutcomeEngine + chapterProgress`，不再依赖 `evaluateDebugChapterOutcome + plan.chapterOutcome`
- `任务 7` 已完成当前阶段：正式/调试结果判定已统一收口到共享服务层，不再由 `SessionService` 和调试路由分别拼装 fallback
- `任务 8` 已推进一阶段：`stateDelta` 已进入白名单模式，编排模型不再能直接改 `turnState / chapterProgress / 用户/旁白/NPC / flags`
- `任务 8` 已继续推进：对外 `plan` 结构已去掉 `chapterOutcome / nextChapterId`，运行链只保留 `speaker / motive / turn flow`
- `任务 8` 已完成当前阶段：正式链对外 `narrativePlan` 和开场落库元数据也已去掉 `chapterOutcome / nextChapterId`
- `任务 9` 已完成当前阶段：Web / Android 编辑器已支持 `runtimeOutline / phase graph` 高级配置输入和阶段预览
- `任务 9` 已继续推进：Web / Android 编辑器已支持“生成草稿 / 格式化 JSON”，不再要求手写完整 phase graph
- `任务 9` 已继续推进：Web / Android 调试界面已可直接查看当前阶段、当前目标、当前用户节点和已完成事件，不再只能盯原始 JSON
- `任务 9` 已继续推进：Web / Android 编辑器已补“阶段关系图”预览，可直接查看 `前置 / 完成 / 分支 / 默认流向 / 关联结果`
- 当前结论：当前这版 `编排师 2.0` 已完成；后续若继续推进，属于拖拽式编辑器和调试体验增强，而不是主链缺失

本拆分默认分三期：
- P1：先把章节推进和用户节点做稳
- P2：把章节结束判定从编排师中剥离
- P3：把章节内容结构化与可视化

---

## P1 总目标
P1 不追求“一步到位最完美”，只要做到：
- 当前章节推进有明确状态
- 用户节点会稳定命中
- 正式模式不再完全靠模型自由发挥
- 编排师只负责局部调度

P1 完成标准：
- 用户节点该轮到用户时，系统一定停下来
- 用户节点没到时，系统不会无故一直等用户
- 自动推进不会越过章节阶段
- 编排师输出不再直接决定章节成功失败
- 编排模型不再直接改关键运行态
- 对外运行计划只保留最小必要字段

当前状态：
- 上述 `P1` 完成标准已满足

## 任务 1：新增 chapterProgress 运行态

### 目标
在会话 state 中引入章节推进状态。

### 新增字段
建议新增：

```json
{
  "chapterProgress": {
    "phaseId": "",
    "phaseIndex": 0,
    "userNodeId": "",
    "userNodeIndex": -1,
    "userNodeStatus": "idle",
    "completedEvents": [],
    "pendingGoal": "",
    "fixedOutcomeLocked": false,
    "lastEvaluatedMessageId": 0
  }
}
```

### 要改的文件
- `src/lib/gameEngine.ts`

### 要新增的函数
- `normalizeChapterProgressState()`
- `readChapterProgressState()`
- `setChapterProgressState()`

### 验收
- 新旧会话状态都能正常归一化
- 缺失字段时自动补默认值
- 不影响已有 `turnState`

## 任务 2：章节运行模板最小版

### 目标
不要再只依赖章节原文，先有一版最小结构化模板。

### 第一版结构
建议在章节对象上先支持：

```json
{
  "runtimeOutline": {
    "openingMessages": [],
    "phases": [],
    "userNodes": [],
    "fixedEvents": [],
    "endingRules": {}
  }
}
```

### 说明
第一版先不用追求 AI 自动提取全部 phase graph。
先能把：
- 开场白
- 最小阶段 `phases`
- 用户节点
- 固定结果
- 成功失败条件
抽出来就够了。

### 当前支持的显式阶段指令
可直接写在某个 `## 阶段标题` 段落内部：

```md
## 萧炎入场
@允许角色: 旁白, 萧炎
@阶段前置: phase:phase_2_user_observe
@阶段完成: fixed_event_xiaoyan_enter
@阶段信号: 萧炎, 看来我来得正是时候
@关联结果: fixed_event_xiaoyan_enter

@萧炎：
"看来我来得正是时候。"
```

说明：
- `@允许角色`：当前阶段允许被编排出来发言的角色名或角色类型
- `@阶段前置`：当前阶段开始前必须已命中的事件
- `@阶段完成`：当前阶段完成时要求命中的事件
- `@阶段信号`：允许推动当前阶段完成的最小文本信号
- `@关联结果`：把当前阶段和固定结果事件显式绑住

### 要改的文件
- `src/routes/game/saveChapter.ts`
- `src/lib/gameEngine.ts`

### 要新增的函数
- `buildChapterRuntimeOutline()`
- `normalizeChapterRuntimeOutline()`

### 验收
- 保存章节时会生成 `runtimeOutline`
- 章节读取时能拿到结构化结果

## 任务 3：新增 ChapterProgressEngine

### 目标
把章节推进从编排师里拆出来，单独交给一个规则引擎。

### 新增文件
- `src/modules/game-runtime/engines/ChapterProgressEngine.ts`

### 职责
- 根据 `runtimeOutline + state + 最新消息` 推进 `chapterProgress`
- 判断当前 phase
- 判断是否命中用户节点
- 标记关键事件完成
- 标记固定结果是否触发

### 输出
建议输出：

```json
{
  "chapterProgress": {},
  "triggeredEvents": [],
  "enteredUserNode": false,
  "pendingGoal": "",
  "shouldYieldToUser": false
}
```

### 要接入的地方
- `src/modules/game-runtime/services/SessionService.ts`
- `src/routes/game/startSession.ts`

### 验收
- 每次用户发言后，章节推进状态会被更新
- 开场自动推进时，章节推进状态也会更新

## 任务 4：让 startSession 初始化 chapterProgress

### 目标
首次进入章节时，不只是创建会话，还要初始化章节推进状态。

### 要改的文件
- `src/routes/game/startSession.ts`

### 要补的逻辑
- 初始化 `chapterProgress`
- 根据 `runtimeOutline` 决定当前 phase
- 判断开场后是否应立即等待用户
- 判断开场后是否可自动推进一小段

### 预期行为
- 首次进第 1 章时，开场几句稳定
- 如果章节开场就是用户节点，则直接等待用户
- 如果章节还有前置 NPC 推进，则先推进到用户节点再停

### 验收
- 首次进入速度不明显变慢
- 首次进入不会再随机停在错误节点

## 任务 8：固定事件与用户影响分离

### 目标
把“固定结果”和“用户差异影响”分开。

### 例子
章节固定结果：
- 徐阳偷走如意金箍棒

用户差异影响：
- 是否受伤
- 是否被注意
- 是否获得线索
- 是否留下标记

### 要改的文件
- `src/modules/game-runtime/engines/ChapterProgressEngine.ts`
- `src/modules/game-runtime/engines/ChapterOutcomeEngine.ts`

### 验收
- 固定结果稳定发生
- 用户行为只影响结果细节，不影响章节主骨架

## P1 实施顺序建议

### 顺序
1. 任务 1：`chapterProgress`
2. 任务 2：`runtimeOutline`
3. 任务 3：`ChapterProgressEngine`
4. 任务 4：`startSession` 初始化
5. 任务 5：改造自动推进
6. 任务 6：收紧编排师输出
7. 任务 7：正式版章节判定器
8. 任务 8：固定事件分离
9. 任务 9：调试/正式统一

### 为什么这样排
- 先有状态
- 再有结构化章节
- 再有推进引擎
- 最后才收掉编排师的权力

这样风险最小。
