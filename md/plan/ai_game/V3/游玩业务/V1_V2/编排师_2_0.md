# 编排师2.0 的设计

## 目标
解决 1.0 中“剧情乱飞”的核心问题：
- 脱离章节内容
- 脱离章节成功条件/失败条件
- 脱离用户交互节点
- 要么用户一直说，要么用户一直没机会说
- 最近聊天记录权重过大，导致章节主线被冲散
- 模型既负责编排又负责判定结束，职责混乱

编排师 2.0 的核心方向不是继续堆提示词，而是：
- 把章节内容从“软提示”改成“硬约束”
- 把章节推进从“自由聊天”改成“受状态机约束的剧情推进”
- 把“谁说话”和“章节是否完成”拆成两个不同职责

---

## 1. 1.0 为什么会乱飞

### 1.1 章节内容只是参考，不是执行骨架
当前章节内容本质上是一整段自然语言提纲。
编排师虽然能读到它，但它没有被拆成：
- 当前阶段
- 当前应触发的事件
- 当前必须出现的用户节点
- 当前距离章节结局还差什么

所以模型只能“参考”，不能“执行”。

这会导致：
- 模型更容易顺着最近几轮对话继续聊
- 而不是沿着章节结构推进

### 1.2 章节结束条件没有变成程序规则
例如章节成功条件：
`徐阳偷走如意金箍棒`

在 1.0 里，这种条件大多只是：
- 放在章节文本里
- 或交给编排师自己理解

问题是：
- 编排师可能忘记
- 可能提前宣布成功
- 可能一直不触发
- 可能被局部闲聊带跑偏

结论：
章节成功/失败条件不能只交给编排师自然语言理解，必须变成结构化规则。

### 1.3 用户节点没有“命中游标”
像“用户行动1 / 用户行动2 / 用户行动3”这种内容，在 1.0 里只是文本说明，不是运行态中的明确节点。

所以系统无法稳定回答这些问题：
- 当前是否已经推进到用户行动1
- 当前是不是必须交给用户
- 当前用户节点已经完成没有
- 当前是不是已经错过用户节点

结果就是：
- 要么过早把回合交给用户
- 要么用户节点已经到了，NPC 还在继续说

### 1.4 编排师职责过大
1.0 的编排师同时负责：
- 谁说话
- 为什么说
- 什么时候轮到用户
- 是否章节完成
- 是否进入下一章
- 甚至隐式影响状态变化

这会让模型变成“既当导演，又当判定器，又当流程机”。
职责一旦过大，任何一个地方理解偏了，整轮都会偏。

---

## 2. 编排师2.0 的核心原则

### 2.1 编排师不再决定大方向，只负责局部调度
编排师只负责：
- 这一步谁说话
- 这一步说话的动机是什么
- 这一步结束后是否该轮到用户

编排师不再负责直接决定：
- 当前章节阶段是什么
- 章节是否成功/失败
- 是否允许跳过关键事件
- 固定结果是否发生
- 是否切换下一章节

### 2.2 章节推进必须有状态机
每个章节都要有一个明确的运行态骨架：
- 当前阶段 `phase`
- 当前用户节点 `userNode`
- 已完成关键事件 `completedEvents`
- 当前章节目标 `activeGoal`
- 当前固定结果是否已锁定 `fixedOutcomeLocked`

编排师只能在当前阶段允许的范围内调度角色发言。

### 2.3 章节结束判定必须独立
章节是否成功/失败不能由编排师自由脑补，必须由独立的章节判定器处理。

职责拆分：
- 编排师：谁说、为什么说
- 发言器：把“角色+动机”写成台词
- 章节判定器：是否满足成功/失败/下一章条件
- 状态机：当前阶段允许发生什么

### 2.4 用户节点必须是硬节点
如果章节设计里明确写了：
- 用户行动1
- 用户行动2
- 用户行动3

那这些不能只是文本说明，而必须是运行态里的真实节点。

一旦命中节点：
- 系统必须停止自动编排
- 必须进入等待用户输入
- 不允许 NPC 抢用户回合

## 3. 编排师2.0 的总体架构

### 3.1 四层职责

#### 第一层：章节结构化编译器
故事发布时，或者章节保存时，把章节自然语言内容编译成结构化章节运行模板。

输入：
- 章节正文
- 开场白
- 成功条件
- 失败条件

输出：
- `runtimeOutline`
- `phaseGraph`
- `userNodes`
- `fixedEvents`
- `endingRules`

#### 第二层：章节状态机
会话运行时维护当前章节推进状态。

作用：
- 判断当前处于哪个阶段
- 判断当前可否自动推进
- 判断当前是否必须等待用户
- 判断当前可以触发哪些事件

#### 第三层：编排师
只接收结构化状态，不接收整章大段自由文本。

只输出：
- `speaker`
- `roleType`
- `motive`
- `awaitUser`
- `nextRole`
- `nextRoleType`

#### 第四层：章节判定器
独立检查：
- 是否满足成功条件
- 是否满足失败条件
- 是否进入下一章节
- 是否进入自由剧情

## 4. 章节结构化设计

### 4.1 章节运行模板
建议每个章节生成一份结构化模板：

```json
{
  "chapterId": 14,
  "title": "第 2 章",
  "opening": [
    { "role": "旁白", "type": "narrator", "content": "混沌虚空之中..." }
  ],
  "phases": [
    {
      "id": "phase_1_battle_start",
      "label": "双悟空交战",
      "mustEvents": ["goku_enter", "battle_started"],
      "allowedSpeakers": ["旁白", "西游孙悟空", "龙珠孙悟空"],
      "targetSummary": "建立混沌战场与初始冲突"
    },
    {
      "id": "phase_2_user_observe",
      "label": "用户行动1",
      "userNodeId": "user_turn_1",
      "allowedSpeakers": ["旁白", "系统"],
      "targetSummary": "把用户推到混沌晶石互动点"
    },
    {
      "id": "phase_3_xiaoyan_join",
      "label": "萧炎入场",
      "mustEvents": ["xiaoyan_enter"],
      "allowedSpeakers": ["旁白", "萧炎"],
      "targetSummary": "局势升级"
    }
  ],
  "userNodes": [
    {
      "id": "user_turn_1",
      "promptRole": "系统",
      "goal": "让用户围绕混沌晶石作出第一次行动",
      "suggestions": ["观察", "接触晶石", "接近战场", "提醒某人"],
      "minInputs": 1
    }
  ],
  "fixedEvents": [
    {
      "id": "xuyang_steals_staff",
      "label": "徐阳偷走如意金箍棒",
      "requiredBeforeFinish": true
    }
  ],
  "endingRules": {
    "success": ["xuyang_steals_staff"],
    "failure": [],
    "nextChapterId": 15
  }
}
```

### 4.2 结构化的价值
这样以后编排师面对的不再是：
- 一大段章节内容

而是：
- 当前 phase
- 当前目标
- 当前允许角色
- 当前还未完成的关键事件
- 当前是否命中用户节点

这样剧情才不会乱飞。

## 5. 会话运行态设计

### 5.1 新增章节推进字段
建议在会话动态数据里增加：

```json
{
  "chapterProgress": {
    "phaseId": "phase_2_user_observe",
    "phaseIndex": 1,
    "userNodeId": "user_turn_1",
    "userNodeStatus": "waiting_input",
    "completedEvents": ["goku_enter", "battle_started"],
    "pendingGoal": "让用户围绕混沌晶石作出第一次行动",
    "fixedOutcomeLocked": false
  }
}
```

### 5.2 turnState 继续保留，但只负责“轮到谁”
`turnState` 的职责保留：
- 当前能否用户发言
- 当前预期说话人是谁

但它不再承担：
- 章节推进判断
- 用户节点推进判断
- 章节结束判断

这些全部交给 `chapterProgress` + `章节判定器`。

## 6. 编排师2.0 输入设计

### 6.1 输入必须缩成结构化最小集
编排师输入不再以“大段提纲文本”为中心，而是以“当前运行态” 为中心。

建议输入：
- 当前世界简介摘要
- 当前章节标题
- 当前 phase 摘要
- 当前用户节点摘要
- 当前章节目标摘要
- 当前已完成事件
- 当前待完成关键事件
- 当前 turnState
- 最近 4 到 8 条强相关对话
- 用户最新输入
- 当前角色列表精炼文本

### 6.2 不再让最近聊天记录压过章节目标
最近对话只作为“局部上下文”，不能成为剧情主驱动。

优先级应该改成：
1. 当前用户节点
2. 当前章节阶段目标
3. 当前待完成关键事件
4. 用户最新输入
5. 最近对话
6. 全局背景

不是现在这种隐式上变成：
1. 最近对话
2. 用户最新输入
3. 章节提纲参考

---

## 7. 编排师2.0 输出设计

### 7.1 只输出局部决策
编排师只输出：

```json
{
  "speaker": "旁白",
  "roleType": "narrator",
  "motive": "把用户推到混沌晶石互动点",
  "awaitUser": false,
  "nextRole": "系统",
  "nextRoleType": "system"
}
```

### 7.2 不允许输出这些内容
编排师不允许直接决定：
- `chapterOutcome`
- `nextChapterId`
- `phaseIndex`
- `completedEvents`
- `fixedOutcome`

这些都应该由规则引擎/判定器算出来。

原因很简单：
这些都是“流程控制字段”，不是“语言模型擅长的自由生成字段”。

---

## 8. 章节判定器设计

### 8.1 单独的判定输入
章节判定器输入：
- 当前章节结构化模板
- 当前章节推进状态
- 最近相关消息
- 最新状态变量
- 用户最近一次行为

### 8.2 判定器输出
判定器输出：

```json
{
  "result": "continue",
  "matchedRules": [],
  "nextChapterId": null,
  "missingGoals": ["xuyang_steals_staff"]
}
```

或者：

```json
{
  "result": "success",
  "matchedRules": ["xuyang_steals_staff"],
  "nextChapterId": 15,
  "missingGoals": []
}
```

### 8.3 固定结果必须程序锁定
像这个例子中：
`徐阳偷走如意金箍棒`

属于章节固定结果。

应该设计成：
- 不是让模型自由决定“偷没偷”
- 而是阶段推进到指定位置后，由规则引擎触发 `xuyang_steals_staff`
- 用户的影响体现在：
  - 是否被徐阳察觉
  - 是否获得标记
  - 是否受伤
  - 是否记录信息

而不是体现在“能不能让固定结局不发生”

---

## 9. 用户节点设计

### 9.1 用户节点是章节推进的闸门
每个用户节点都要像关卡一样。

进入用户节点后：
- `awaitUser = true`
- 禁止自动继续推进下一 phase
- 必须等待用户输入或用户跳过

### 9.2 用户输入后的处理
用户输入后不是直接自由乱编，而是：
1. 解析用户行为
2. 更新状态
3. 标记用户节点完成
4. 推进到下一阶段
5. 再让编排师只负责编排下一小步

### 9.3 用户跳过也是显式行为
用户发送 `.` 时，不是普通文本，而是：
- 当前用户节点选择跳过
- 由规则引擎决定跳过后的后果

而不是简单让编排师自由发挥一句“你沉默了，别人继续说”。

---

## 10. 发言器设计

### 10.1 发言器只负责落地当前一步
发言器输入：
- 说话角色
- 当前动机
- 当前阶段
- 最近少量对话
- 用户最新输入

发言器输出：
- 当前这一轮真正展示给用户的台词

### 10.2 发言器不能跨步
发言器禁止：
- 提前把后续剧情讲完
- 一口气把整个 phase 讲完
- 越过用户节点
- 把章节内部提纲念出来

发言器要做的是：
只完成这一小步。

---

## 11. 状态修改边界

### 11.1 模型允许修改的状态
编排师/发言器最多只能提出建议性的轻状态变化，例如：
- 情绪变化
- 态度变化
- 临时关注点
- 轻量记忆提示

### 11.2 模型禁止直接修改的状态
以下必须由规则引擎修改：
- 当前 phase
- 用户节点是否完成
- 关键事件是否完成
- 章节成功/失败
- 下一章节 id
- 固定结果是否发生

这样才能防止状态漂移。

## 12. 第2章示例：2.0 应该如何运行

### 12.1 初始阶段
进入章节后：
- phase = `phase_1_battle_start`
- 已完成事件为空
- 系统自动播开场与悟空交战内容

### 12.2 推进到用户行动1
当：
- `goku_enter`
- `battle_started`

都完成后，状态机切到：
- `phase_2_user_observe`
- `userNodeId = user_turn_1`
- `userNodeStatus = waiting_input`

此时必须：
- 编排系统停止自动推进
- 等待用户输入

### 12.3 用户行动1完成
用户输入后：
- 解析成观察 / 接触 / 接近 / 提醒等行为
- 写入状态
- `user_turn_1 = done`
- phase -> `phase_3_xiaoyan_join`

### 12.4 固定结局
推进到高潮时：
- `time_stop`
- `user_turn_3`
- `xuyang_steals_staff`

最终固定结果一定发生：
- 如意金箍棒被夺

但用户差异影响可以不同：
- 是否获得徐阳标记
- 是否受伤
- 是否记录线索
- 是否被徐阳注意

这才符合“既有固定章节结果，又允许用户影响细节”。

## 13. 接口调整建议

### 13.1 `/orchestration`
职责改为：
- 基于当前 `chapterProgress + turnState` 决定下一步谁说、为什么说

返回：
- `speaker`
- `roleType`
- `motive`
- `awaitUser`
- `nextRole`
- `nextRoleType`

### 13.2 `/streamlines`
职责不变：
- 把当前 speaker + motive 写成可展示正文

### 13.3 新增 `/chapter-evaluate`
职责：
- 对当前章节做独立成功/失败/继续判定

### 13.4 新增 `/chapter-progress`
职责：
- 根据最新一轮用户输入或最新事件，推进 `phase/userNode/completedEvents`

也可以不拆成独立接口，但服务内一定要拆成独立模块。

## 14. 数据持久化建议

### 14.1 章节静态数据
故事发布时预生成：
- 章节结构化模板
- 用户节点列表
- 关键事件列表
- 固定结果列表
- 结束条件规则

### 14.2 对话动态数据
对话持久化时保存：
- `chapterProgress`
- `turnState`
- `story dynamic state`
- `role dynamic state`
- 最近一次已处理消息游标

这样续聊时才能真正快且稳定。

---

## 15. 编排师2.0 的最终职责定义

### 15.1 编排师负责
- 在当前阶段内选择合适角色
- 给出这一轮推进动机
- 判断这一轮结束后是否该轮到用户

### 15.2 编排师不负责
- 判定章节成功失败
- 决定固定结局是否发生
- 决定是否跳 phase
- 决定是否切下一章
- 直接修改关键状态

---

## 16. 分阶段落地建议

### 第一阶段
- 增加 `chapterProgress`
- 把用户节点做成硬节点
- 编排师只保留 `speaker + motive + awaitUser`

### 第二阶段
- 引入正式版章节判定器
- 正式模式不再依赖编排师返回 `chapterOutcome`

### 第三阶段
- 章节内容预编译成结构化模板
- 固定事件、主线事件、用户节点全部结构化

### 第四阶段
- 记忆管理师只做背景/角色/摘要维护
- 不再承担章节推进核心职责

---

## 17. 一句话总结
编排师 1.0 的问题不是“模型不聪明”，而是“职责设计错误”。

1.0 是：
- 用一个大模型同时负责剧情编排、角色调度、章节判定、阶段推进

2.0 应改成：
- 状态机负责章节推进
- 判定器负责章节结束
- 编排师只负责这一小步谁说、为什么说
- 发言器只负责把这一小步写成自然台词

这样才能做到：
- 不脱离章节
- 不脱离用户
- 不脱离结束条件
- 同时仍然保持编排足够快

### 18.1 第一阶段目标
先不要一口气把所有东西重构完。

第一阶段只做三件事：
- 把章节用户节点做成硬节点
- 把章节推进状态独立出来
- 把编排师的职责缩回到“局部调度”

也就是说先解决最痛的三个问题：
- 用户该说话时却没机会说
- 用户不该说话时却一直让用户说

### 21.3 当前代码应修改的位置

## 23. 独立章节判定器

## 24. 固定事件与用户影响分离

### 24.1 当前问题
像“徐阳偷走如意金箍棒”这种章节固定结果，在 1.0 里会被模型当成普通叙事内容处理。

结果就是：
- 有时忘记发生
- 有时提前发生
- 有时被用户行为硬拐没了

### 24.2 2.0 的设计
固定事件必须由规则引擎锁定。

例如：
- `xuyang_steals_staff`
属于固定事件

用户的影响不作用于“事件是否发生”，而作用于：
- 发生时附带什么代价
- 用户获得什么线索
- 用户和徐阳建立什么关系

### 24.3 当前代码应修改的位置

#### `src/modules/game-runtime/services/SessionService.ts`
在每次落库新消息、提交 narrative turn 后：
- 触发 `RuntimeProgressEngine`
- 判断是否要打上固定事件完成标记

建议新增一个新模块：
- `src/modules/game-runtime/engines/ChapterProgressEngine.ts`

职责：
- 推进 `chapterProgress`
- 标记 `completedEvents`
- 判断是否命中固定事件

## 27. 记忆管理师的边界

### 27.1 当前原则
记忆管理师继续保留，但边界要缩清楚。

记忆管理师负责：
- 背景总结
- 角色动态参数摘要
- 关系变化摘要
- 世界状态摘要

记忆管理师不负责：
- 决定当前章节推进到哪
- 决定是否结束章节
- 决定当前是否该轮到用户

### 27.2 当前代码应调整的位置

#### `src/modules/game-runtime/engines/NarrativeOrchestrator.ts`
`triggerMemoryAgent` 仍可保留，但它只是后台记忆刷新信号，不应再承担章节流程推进含义。
