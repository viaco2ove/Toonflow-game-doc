## 目标
在 2.0 “剧情不乱飞”的基础上，继续解决另一个核心问题：

- 角色出词太慢
- 每轮模型链路太长
- token 消耗过高
- 编排师、发言器、记忆管理经常一起烧
- 前端容易因为长时间无返回而误报超时
- 很多轮次明明是确定流程，却仍然重复走完整模型链

3.0 的目标不是单纯“再压一点 prompt”，而是：

- 让大部分轮次尽量接近秒回
- 减少每轮必须调用的大模型次数
- 让“规则引擎优先，模型补细节”真正落地
- 把高质量生成只留给关键角色和关键时刻
- 在尽量少提示词的前提下，仍然保留动态背景、动态参数、动态关系

一句话：

**2.0 解决“别乱飞”，3.0 解决“别太慢、别太贵”。**

## 1. 为什么 2.0 之后依然慢

2.0 已经把章节推进、用户节点、阶段约束、结束判定做稳了。
但当前每轮实际仍然可能是：

1. 编排师判断谁说话
2. 角色发言器生成正文
3. 记忆管理器抽取摘要
4. 语音模型生成语音

在远程模型场景下，这几步任何一步慢，整轮就慢。

### 1.1 真正最慢的通常不是编排师，而是角色发言器
从现有日志看：

- 编排师经常是 7~10 秒
- 角色发言器也经常是 7~15 秒，甚至更高
- 如果模型带强 reasoning，还会出现高额 `reasoning_tokens`

这意味着：

- 单纯缩编排师 prompt 还不够
- 真正影响体感的是“谁说话之后怎么快速落出一句自然台词”

### 1.2 很多轮次本来不需要模型决定“谁说”
例如：

- 当前 phase 已经强约束为 `某男子 -> 萧炎 -> 旁白`
- 当前轮次只是承接上一句继续挑衅
- 当前只是环境氛围补一句
- 当前只是明确的用户节点前一拍

这类轮次如果还让编排师重新读完整上下文再决定谁说，本质上是浪费。

### 1.3 当前 speaker 任务对模型来说仍然“太重”
即使 2.0 已经做过一次瘦身，speaker 当前仍然会看到：

- 当前角色资料
- 当前阶段
- 动机
- 故事状态摘要
- 最近对话
- 其他可见角色

这已经比 1.0 好很多，但对于“只生成一句台词”来说，仍然偏重。

### 1.4 记忆管理更新过于频繁
如果每轮都刷新记忆：

- 会增加 token
- 会增加后台请求
- 会让一次普通对话变成“三模型并发烧钱”

而事实上，不是每轮都值得更新记忆。

### 1.5 前端/移动端把“长时间无输出”当成失败
如果 `streamlines` 在模型真正出字前完全无流式输出，前端就会误以为超时。

这会导致：

- 前端先报错
- 后端稍后成功
- 页面状态看起来和后端不一致

所以 3.0 不只是“减少 token”，还要做“感知层面的快”。

## 2. 3.0 的核心原则

### 2.1 规则优先，不必每轮都问模型
凡是规则引擎能确定的，不交给模型：

- 当前阶段该谁先说
- 当前阶段是否必须先让用户观察
- 当前是否进入用户节点
- 当前是否只需要环境补一句

模型只做规则无法确定、或者需要自然语言表达质量的部分。

### 2.2 发言分级，不是所有角色都配同样成本
不同角色/不同轮次走不同档位：

- 模板直出
- 轻量模型
- 高质量模型

不是所有角色都必须走高质量远程模型。

### 2.3 先让用户看到结果，再补细节
如果“高质量正文”需要几秒，先给用户一个低延迟结果：

- 简短台词
- 占位式场景反应
- 极简动作句

然后后台再补长一点的正文或语音。

### 2.4 动态背景和动态参数保留，但只保留“当前轮真正需要的那部分”
3.0 不是砍掉动态能力，而是把动态能力做成“按需加载”：

- 不再把整段世界背景塞给发言器
- 不再把整份参数卡塞给每一轮
- 只把当前 speaker 真正需要的 3~6 个动态要点塞进去

### 2.5 记忆更新从“每轮刷新”改成“按事件刷新”
只有命中关键变化时才触发记忆更新：

- 新关系建立
- 关键道具变化
- 新任务
- 阶段切换
- 章节判定
- 用户关键抉择

普通插科打诨不刷新。

## 3. 3.0 的总体架构

3.0 把当前链路拆成五层：

### 第一层：章节状态机 / phase graph
职责不变，继续由 2.0 提供：

- 当前阶段
- 当前允许角色
- 当前目标
- 当前用户节点
- 当前完成事件

这是所有后续决策的基础。

### 第二层：规则编排器
这是 3.0 新增重点。

职责：

- 尽量直接根据 phase graph 得出下一位 speaker
- 只在规则无法确定时才调用 AI 编排师

输出：

- `speaker`
- `roleType`
- `motive`
- `awaitUser`
- `nextRole`
- `nextRoleType`
- `decisionSource`

其中 `decisionSource`：

- `rule`
- `ai`
- `fallback`

### 第三层：发言策略路由器
根据当前轮次决定走哪种发言档位。

输出：

- `mode: template | fast | premium`
- `voiceMode: skip | async | immediate`
- `memoryMode: skip | async`

### 第四层：角色发言器
只负责把：

- `speaker`
- `motive`
- `recentDialogueLite`
- `dynamicContextLite`

写成用户可见正文。

### 第五层：后处理器
只负责：

- 异步记忆刷新
- 异步语音
- 富化正文
- 更新会话摘要

不阻塞首屏返回。

## 4. 三档发言机制

### 4.1 T0：模板直出
适用：

- 旁白环境补句
- 万能角色/路人起哄
- 明显功能性推进句
- 已有强规则的阶段过渡句

特点：

- 不调用大模型
- 直接根据 `speaker + motive + phase` 用模板拼装
- 目标是 50~200ms 内出结果

示例：

- 旁白：
  - “(练武场的哄笑声仍未停歇，几道目光在你与萧炎之间来回扫动)”
- 某男子：
  - “(抱着胳膊冷笑一声) 怎么，不敢接话了？”

### 4.2 T1：轻量模型
适用：

- 普通 NPC
- 泛旁白
- 万能角色临演
- 非关键角色的过场发言

特点：

- 用低延迟模型
- 严格短上下文
- 目标 1~3 秒

### 4.3 T2：高质量模型
适用：

- 萧炎、纳兰嫣然、薰儿等关键角色
- 用户首次建立关系
- 关键冲突
- 高价值章节节点

特点：

- 允许更高质量
- 允许更长一点的动态上下文
- 但仍必须先有心跳或首包占位

## 5. 规则编排器设计

### 5.1 什么时候可以直接绕过 AI 编排师
满足以下任一条件时，规则直接给出下一位 speaker：

1. 当前 phase 明确只有一个允许角色
2. 当前 phase 的默认流向已经固定，且上一轮刚好命中其完成信号
3. 当前轮是固定模板路人/旁白补一句
4. 当前已经到用户节点，必须 `awaitUser=true`
5. 当前轮只是“补一个环境反应”而非真实决策

### 5.2 什么时候才调用 AI 编排师
只在这些情况下调用：

1. 当前 phase 允许多个角色，且规则无法决定谁更自然
2. 当前轮涉及用户态度、阵营、关系、任务走向等不确定性
3. 当前轮需要在多个关键角色之间决定谁接话
4. 当前轮需要根据近几轮动态局势决定“谁更适合推进”

### 5.3 规则编排器的输出
建议新增结构：

```json
{
  "role": "某男子",
  "roleType": "npc",
  "motive": "继续挑衅萧炎，同时逼迫异天表态",
  "awaitUser": false,
  "nextRole": "萧炎",
  "nextRoleType": "npc",
  "decisionSource": "rule"
}
```

这样前端、日志和统计都能直接看到：

- 这轮是不是 AI 生成的
- 哪些轮次其实根本没必要烧编排模型

## 6. speaker 极简上下文设计

### 6.1 当前 speaker 输入仍然过重的问题
当前 speaker 虽然已经压缩，但仍然会带：

- 世界简介
- 角色参数卡压缩版
- 故事状态摘要
- 最近多轮对话
- 其他可见角色

这对“出一句话”来说仍然偏重。

### 6.2 3.0 的 speaker 输入原则
speaker 只保留五块：

1. 当前说话人最小资料
2. 当前动机
3. 当前阶段标签
4. 最近 2~3 条必要对话
5. 当前用户最新输入

不要再默认带：

- 长世界简介
- 长故事背景摘要
- 大段角色参数卡
- 超过 3 条以上最近对话
- 大量“其他可见角色”

### 6.3 最小 speaker payload
建议像这样：

```txt
[speaker]
name: 萧炎
role_type: npc
traits: 隐忍、屈辱、少年心性强、嘴硬

[phase]
label: 场景背景

[motive]
屈辱已达极限，压着怒火回怼带头挑衅的萧家子弟

[recent]
某男子：……
旁白：……

[player]
无
```

目标：

- 能写出自然台词
- 不再让模型把注意力浪费在世界观复述上

## 7. 动态背景与动态参数如何保留

### 7.1 不是删除动态，而是拆成三层缓存

#### 第一层：角色静态摘要缓存
每个角色提前生成：

- `roleLiteProfile`

例如：

```json
{
  "name": "萧炎",
  "roleType": "npc",
  "traits": ["隐忍", "少年天才跌落谷底", "重尊严"],
  "speechStyle": "压抑、克制、偶尔锋利"
}
```

这份内容在发言时直接取，不再每轮重新压参数卡。

#### 第二层：会话动态摘要缓存
每个会话维护：

- 当前关系变化
- 当前任务
- 当前可见冲突
- 当前重要道具变化

只保留最关键的 3~5 条。

#### 第三层：当前轮动态焦点
每轮临时计算：

- 当前阶段目标
- 当前冲突焦点
- 当前 speaker 真正关心的那件事

speaker 只读第三层，必要时附带第二层的一小部分。

### 7.2 这样做的结果
仍然保留：

- 动态背景
- 动态人物状态
- 动态关系
- 动态任务

但不需要每轮把整份状态重讲一遍。

## 8. 记忆管理 3.0

### 8.1 只按事件更新，不按轮次更新
触发条件建议收敛为：

- 新关系建立/破裂
- 用户做出明确立场选择
- 新任务出现
- 新关键道具出现/消失
- 章节阶段切换
- 章节成功/失败
- 关键角色状态变化

其他情况：

- 不刷新记忆
- 只在本地状态里记最近消息

### 8.2 记忆更新一律后台异步
首屏返回不要等记忆管理完成。

发言成功后：

- 先提交消息
- 后台排队做记忆更新
- 成功后静默更新 `memorySummary`

## 9. 语音链路 3.0

### 9.1 语音不能阻塞正文首屏
正文必须先落。

语音统一分两种：

- `immediate`
  - 当前消息刚落库就异步请求语音
- `deferred`
  - 用户停留时再请求

### 9.2 旁白/万能角色优先走标准音色
因为这类内容本来就是低成本内容：

- 不需要每轮都跑复杂克隆链
- 优先走标准预设音色
- 进一步减少等待

## 10. 为了“秒回”要改的真实链路

### 10.1 当前链路

```txt
用户输入
-> addMessage
-> orchestration
-> streamlines
-> commitNarrativeTurn
-> memory
-> voice
```

### 10.2 3.0 目标链路

```txt
用户输入
-> addMessage
-> ruleOrchestrate
   -> 若规则能决定：直接出 plan
   -> 若规则不能决定：才走 ai orchestration
-> routeSpeakerMode
   -> template / fast / premium
-> 先返回正文
-> commitNarrativeTurn
-> memory async
-> voice async
```

### 10.3 关键原则
首屏感知时间只看：

- `addMessage`
- `ruleOrchestrate / ai orchestration`
- `template/fast/premium speaker`

记忆和语音都不能阻塞“先看到字”。

## 11. 3.0 的最小接口设计

### 11.1 规则编排器接口

建议新增：

```ts
type RuleNarrativePlan = {
  role: string;
  roleType: "narrator" | "npc" | "player";
  motive: string;
  awaitUser: boolean;
  nextRole: string;
  nextRoleType: "narrator" | "npc" | "player";
  decisionSource: "rule" | "ai" | "fallback";
  confidence: "high" | "medium" | "low";
  reason: string;
};
```

说明：

- `decisionSource` 用于日志、统计和调试面板
- `confidence` 用于决定是否回退给 AI 编排师
- `reason` 只用于调试，不展示给用户

### 11.2 发言策略路由器接口

```ts
type SpeakerModeDecision = {
  mode: "template" | "fast" | "premium";
  voiceMode: "skip" | "async" | "immediate";
  memoryMode: "skip" | "async";
  reason: string;
};
```

说明：

- `template` 主要给旁白、万能角色、功能性补句
- `fast` 给普通 NPC 和普通过场
- `premium` 只给关键角色与关键节点

### 11.3 极简 speaker 输入接口

```ts
type SpeakerLiteInput = {
  speaker: {
    name: string;
    roleType: "narrator" | "npc";
    traits: string[];
    speechStyle: string;
  };
  phase: {
    id: string;
    label: string;
  };
  motive: string;
  recentDialogueLite: string[];
  currentUserInput: string;
  dynamicFocus: string[];
};
```

要求：

- `recentDialogueLite` 默认 2~3 条
- `dynamicFocus` 默认 2~4 条
- 禁止直接灌整份参数卡和整段世界背景

### 11.4 动态缓存接口

```ts
type SessionDynamicDigest = {
  currentPhaseId: string;
  currentConflict: string;
  currentGoal: string;
  keyFacts: string[];
  relationChanges: string[];
  taskChanges: string[];
  itemChanges: string[];
  visibleFocusRoles: string[];
  updatedAt: number;
};
```

说明：

- 这是会话级缓存
- 供编排师、speaker、调试界面复用
- 目的是替代“每轮都重新总结一大段故事状态”

## 12. 3.0 的推荐实施顺序

### P1：先解决“误报超时”和“speaker 人格混乱”

优先级最高：

1. `streamlines` 心跳
2. `speaker` 系统提示词去掉总调度人格
3. `speaker` payload 进一步瘦身

做到这里，哪怕还没减少模型次数，也能明显改善：

- 安卓/Web 不再轻易误报超时
- speaker 推理量下降
- 首屏体感变稳

### P2：减少每轮模型次数

优先实现：

1. `RuleOrchestrator`
2. AI 编排师降级为 fallback
3. 旁白/万能角色的模板化输出

做到这里，很多轮次会从：

```txt
编排模型 + 发言模型
```

变成：

```txt
规则编排 + 模板/轻量发言
```

### P3：发言分级

重点做：

1. `template` 档
2. `fast` 档
3. `premium` 档

这一步决定真正的成本控制能力。

### P4：缓存与异步化

最后做：

1. `roleLiteProfile`
2. `sessionDynamicDigest`
3. 事件触发式记忆管理
4. 语音完全异步

做到这里，3.0 才算真正闭环。

## 13. 3.0 的验收指标

3.0 不能只看“代码写完”，必须看效果。

### 13.1 速度指标

- 模板直出：
  - 目标 200ms 内返回正文
- `fast` speaker：
  - 目标 1~3 秒
- `premium` speaker：
  - 目标 3~8 秒
- 前端长请求误报超时：
  - 降到接近 0

### 13.2 成本指标

- 普通剧情轮次平均模型调用数下降
- 普通剧情轮次平均 token 数下降
- `premium` 只占总发言轮次的小部分

### 13.3 质量指标

- 不出现“角色说错人”
- 不出现“模型代替用户说台词”
- 不出现“章节提纲被直接念给用户”
- 不因为模板化导致明显机械重复

### 13.4 运行态一致性指标

- 前端状态和后端落库状态一致
- `currentStatus` 不再频繁出现“前端失败但后端成功”
- 自动推进在用户节点能稳定停下

## 14. 3.0 落地后的预期变化

如果 3.0 做完，理想中的运行表现应该是：

- 旁白补句、路人起哄、环境过渡，大部分直接秒回
- 普通 NPC 大多在 1~3 秒内出字
- 只有关键角色、关键节点才需要更高质量模型
- 语音和记忆变成后台异步，不再拖慢正文
- 编排师从“每轮都上”变成“规则不确定时才上”

也就是说，用户体感会从：

```txt
每一步都在等 AI 想完
```

变成：

```txt
大部分时候很快，只有关键时刻更讲究
```

## 15. 一句话总结

2.0 的核心是：

- 让剧情不要乱飞

3.0 的核心是：

- 让绝大多数轮次不必每次都重新“完整思考一遍”

所以 3.0 本质上不是“继续抠提示词”，而是：

- 用规则替代一部分编排
- 用分级替代统一高成本发言
- 用缓存替代重复塞上下文
- 用异步替代阻塞链路

只有这样，AI 故事系统才有机会同时做到：

- 稳
- 快
- 省
- 还保留动态背景和动态角色状态

## 12. 3.0 的实施顺序

### P1：先解决“前端感知慢”
1. `streamlines` 心跳
2. 记忆和语音完全异步
3. speaker prompt 再瘦一轮

### P2：减少模型调用次数
1. 新增规则编排器
2. phase 能直接产出 `speaker`
3. 只有规则无法决定时才走 AI 编排师

### P3：发言分级
1. 模板直出
2. fast speaker
3. premium speaker

### P4：动态上下文缓存
1. `roleLiteProfile`
2. `sessionDynamicDigest`
3. `currentTurnFocus`

## 13. 3.0 完成标准

### 体感指标
- 大多数普通轮次 1~3 秒内看到正文
- 关键角色轮次尽量压到 3~6 秒
- 不再频繁出现“前端先报超时，后端后面才成功”

### 成本指标
- 普通轮次不再固定烧“编排师 + 发言器 + 记忆管理”三次调用
- 万能角色/旁白/路人轮次尽量不走高质量模型

### 质量指标
- 仍保持 2.0 的：
  - 不乱飞
  - 不越阶段
  - 不抢用户回合
  - 不脱离章节目标

---

## 14. 结论

2.0 的核心是：

- 把剧情从“自由聊天”拉回“受章节状态机约束”

3.0 的核心则是：

- 把“每轮都要问远程大模型全套链路”改成“规则优先 + 分级生成 + 异步后处理”

真正要实现“角色尽量秒回”，不能继续依赖：

- 每轮都重新编排
- 每轮都重新长提示词写台词
- 每轮都同步更新记忆和语音

而必须变成：

- **规则先决定流程**
- **模板和轻量模型承担大多数发言**
- **高质量模型只服务关键角色和关键时刻**
- **用户先看到字，再补语音和附加处理**

这才是 3.0 的方向。
