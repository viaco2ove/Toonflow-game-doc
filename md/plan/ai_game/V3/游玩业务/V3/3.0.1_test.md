# 3.0.1 自由章节测试说明

## 结论先说

当前推荐测试顺序是：

1. 先做章节调试
2. 调试通过后，再发布并做真实会话测试

原因不是重复劳动，而是这两条链不完全一样：

- 章节调试走的是调试态运行时与 `/game/orchestration`
- 正式游玩走的是 `SessionService` 持久化会话链

所以：

- 调试通过，说明编排链大体成立
- 发布后真实会话再通过，才说明业务闭环成立

- 当前事件摘要为空时

## 现在怎么测

## 第一阶段：章节调试

### 推荐准备两种测试章节

#### A. 纯自由章节

特点：

- `completionCondition = null`
- 无 ending rules
- 静态 phase 可以是 0 个，或者很少

目标：

- 验证零静态事件时是否能直接起一个动态事件
- 验证当前事件为空时，编排师是否会主动生成 `event_summary`

#### B. 半自由章节

特点：

- 先有 1 到 2 个静态 phase
- 没有结束条件

目标：

- 验证静态事件跑完后，是否自动切到新的动态事件

### 调试时重点看什么

章节调试不要只看“有没有一句台词出来”，重点看这几个字段：

- `currentEventDigest`
- `eventDigestWindow`
- `eventDigestWindowText`

现在调试接口已经会直接返回这些数据。

#### 需要重点确认的现象

1. 首次进入章节调试时

- 如果没有静态事件，也应该能出现一个当前事件
- 不应该卡在空 phase

2. 用户发言一轮后

- `currentEventDigest.eventSummary` 应该存在
- `eventStatus` 应该正确变化

3. 静态事件跑完后

- `eventIndex` 应继续增长
- 不应回退到第一个静态 phase
- `dynamicEvents` 应继续保留历史动态事件

4. 章节没有结束条件时

- 调试结果应持续 `continue`
- 不应直接结束

### 章节调试的判定标准

只要下面四项都成立，就算章节调试通过：

1. 不会卡死在空事件
2. 当前事件摘要会持续更新
3. 静态事件耗尽后还能继续推进
4. 事件索引和事件窗口持续前进，不会回退

## 第二阶段：发布后真实会话测试

章节调试通过后，再发布并做正式游玩测试。

原因：

- 正式会话链还要经过 `SessionService`
- 包括：
  - 消息落库
  - 会话状态持久化
  - 后台记忆刷新
  - 真实 turnState 流转

### 正式会话重点验证

1. 连续多轮后，会话仍保持 `active`
2. 无结束条件章节不会错误进入 `chapter_completed`
3. `currentEventDigest` 会持续变化
4. `eventDigestWindowText` 能反映当前事件窗口
5. 用户发送 `.` 跳过后，系统仍能继续推进

## 静态事件与动态事件当前如何生成和管理

## 静态事件

静态事件来自章节内容编译后的：

- `runtimeOutline.phases`

它们负责：

- 初始事件顺序
- 用户交互节点
- phase 目标摘要
- allowedSpeakers 等硬约束

## 动态事件

动态事件放在：

- `state.dynamicEvents`

当前动态事件由：

- `state.currentEvent`
- `currentEventDigest`

共同反映。

### 当前动态事件的生成规则

#### 情况 1：静态 phase 还没跑完

- 当前事件优先跟着 phase 走

#### 情况 2：自由章节且没有静态 phase

- 初始化时直接创建事件 1

#### 情况 3：自由章节且静态 phase 已耗尽

- 自动切到新的动态事件索引
- 当前事件摘要可以先为空
- 下一轮由编排师补出新的 `event_summary / event_facts`

### 当前动态事件的管理规则

当前已经做好的管理能力：

1. 运行时动态事件会保留，不会在同步 phase 时被直接抹掉
2. 统一事件视图已经打通：
   - `currentEventDigest`
   - `eventDigestWindow`
   - `eventDigestWindowText`
3. 事件记忆会跟事件 digest 一起走
4. 前端事件面板已经加了静态事件列表兜底：
   - 如果运行时 `eventDigestWindow` 还只有占位事件
   - 会先展示当前章节 `runtimeOutline.phases` 的静态事件列表和进度

## 目前还没做成什么样

当前“自由章节”后端机制已经基本成立，但还没有这些东西：

1. 前端专门的动态事件可视化管理面板
2. 手动编辑 `dynamicEvents` 的工具化 UI
3. 明确的“自由章节编辑器模式”

所以现在的状态是：

- 后端运行机制：已可测
- 调试链：可测
- 正式会话链：可测
- 前端可视化管理：还不完整

## 当前建议的实际测试法

### 最小测试法

1. 新建一个没有结束条件的测试章节
2. 先在章节调试里跑
3. 看 `currentEventDigest` 和 `eventDigestWindowText`
4. 确认事件会继续推进
5. 再发布后开真实会话
6. 再确认真实会话里不会错误结束

### 推荐重点观察的字段

- `currentEventDigest.eventIndex`
- `currentEventDigest.eventSummary`
- `currentEventDigest.eventStatus`
- `eventDigestWindowText`
- 前端“当前章节事件”面板里展示的是运行时事件窗口还是静态事件兜底列表
- 会话 `status`
- 会话 `chapterId`

## 当前结论

当前自由章节的状态不是“只写了设计”，而是：

- 后端已经具备自由章节运行机制
- 可以进入章节调试验证
- 调试通过后应继续做正式发布测试

所以现在的正确测试路径就是：

**先章节调试，通过后再发布做真实会话测试。**
