# 3.0 bug AI 纠偏方案

## 目标

这次纠偏不是继续给 `3.0` 打补丁，而是把职责重新分清：

- 编排师重新回到主链
- 规则层只保留硬约束，不再代替编排师做大部分业务判断
- 章节内容改为事件主导，而不是让 `phase` 直接承担编排
- 记忆刷新改成显式策略，而不是被 `rule / template / fast` 副作用决定

## 核心判断

当前 `3.0` 的主要问题不是“模型不够强”，而是：

1. `RuleOrchestrator` 命中面过大，抢走了太多本该由编排师处理的轮次
2. `SpeakerRouteEngine` 分级边界过宽，很多关键轮次被降成 `template / fast`
3. 记忆刷新被过度压缩，导致“编排师和记忆管理师像罢工”

所以这次不再继续扩规则，而是反向收权。

## 总体方案

### 1. 编排师重新主导

保留“编排流程(`/game/orchestration`)
编排->下一个角色获取新台词->
返回台词 -> 编排中 + 语音播放中 -> 两件事结束 -> 下一角色拿新台词”这条主链。

但是编排师要做轻量化，不再发送整章大段原文，而是发送：

- 最近 8~10 条对话
- 当前事件摘要
- 当前事件序号
- 当前事件的可说话角色
- 当前角色的文字化精简参数卡
- 当前故事的动态背景摘要

编排师返回：

- 下一个说话角色
- 角色动机
- 当前事件是否需要调整
- 调整后的事件摘要
- 是否触发记忆管理

### 2. 章节内容事件化

章节不再以“phase 直接驱动编排”的方式运行，而是改成“事件主导”。

建议拆成三层：

#### 2.1 发布态静态事件

```ts
{
  storyId,
  chapterId,
  eventIndex,
  eventType,
  eventSummary,
  allowedRoles,
  hardConstraints,
}
```

#### 2.2 会话态动态事件

```ts
{
  storyId,
  chapterId,
  sessionId,
  eventIndex,
  runtimeEventSummary,
  runtimeFacts,
  status,
}
```

#### 2.3 会话事件进度

```ts
{
  storyId,
  sessionId,
  chapterId,
  currentEventIndex,
  currentEventStatus,
}
```

编排师可以调整：

- `runtimeEventSummary`
- `runtimeFacts`
- `currentEventStatus`

但不能随意改整章主干。

### 3. phase 不删除，降级成硬约束壳

`phase` 不应继续承担“直接编排谁说话”，但也不该删掉。

保留 `phase` 的价值仅限于：

- 当前是否允许用户发言
- 当前允许哪些角色发言
- 当前事件属于过场 / 冲突 / 选择 / 收束
- 当前事件是否完成

也就是说：

- `phase` = 硬约束壳
- `event` = 当前剧情单元
- `orchestrator` = 在约束壳内决定下一轮

### 4. 记忆管理显式化

记忆不再“每轮刷新”，也不能再靠 `template / fast / rule` 副作用决定是否刷新。

改成两段式：

#### 4.1 编排师可触发

编排师显式返回：

- `triggerMemoryAgent`
- `memoryReason`

#### 4.2 Worker 只做增量

记忆管理师每 30 秒扫一次，只发送“上次已处理台词序号之后的新台词”。

需要额外记录：

```ts
{
  sessionId,
  lastMemoryMessageId,
  lastMemoryLineIndex,
  updateTime,
}
```

只有当出现新台词时才真正访问大模型。

### 5. 发言分级收紧

当前 `template / fast / premium` 还可以保留，但边界要收回：

- `template`
  - 只允许纯过场、纯氛围、纯功能句
- `fast`
  - 只允许普通 NPC 的低风险短承接句
- `premium`
  - 关键角色冲突
  - 立场变化
  - 关系变化
  - 重要转折
  - 用户明确表态后的回应

尤其是：

- 萧炎
- 纳兰嫣然
- 薰儿
- 小医仙

这类关键角色不能再大面积走 `fast / template`。

## 第一阶段可直接执行的修复

### P0.1 收回规则层权限

`RuleOrchestrator` 只保留两类直出：

1. 当前明确轮到用户
2. 当前 `allowedSpeakers` 真正只剩一个可用角色

删除或停用这些自动短路：

- `expected_role`
- `phase_named_role`
- `phase_wildcard_role`
- `narrator_bridge`

### P0.2 恢复 AI 编排主导

只要不属于上面两类硬直出，就必须走编排师。

### P0.3 收紧 speaker 分级

先把最容易出问题的这几类收回 `premium`：

- 关键角色
- 用户刚刚表态后的回应
- 立场压迫 / 冲突升级 / 关系变化

### P0.4 记忆管理止损

在新事件化方案没落地前，先恢复更积极的记忆触发：

- 用户明确表态
- 新人物关系变化
- 新事件推进
- 当前章节事件摘要变化

## 第二阶段实现

### P1 事件化存储

增加章节静态事件与会话动态事件结构。

### P2 编排输入瘦身

编排师输入从“章节原文 + 大段 phase 提示”改成：

- 最近对话
- 当前事件
- 当前事件摘要
- 精简动态背景
- 精简角色参数

### P3 记忆管理增量化

按 `lastMemoryMessageId / lastMemoryLineIndex` 做真正增量扫描。

## 本次执行结果

这轮纠偏已经不再停留在 `P0`，而是沿着 `P0 -> P1 -> P3` 连续推进完成了后端主链收口：

1. `RuleOrchestrator` 已收回为最小硬直出
2. 编排师、speaker、记忆三条链已统一消费事件 digest
3. 事件视图已统一收口到 `gameEngine`
4. 会话接口、调试接口、调试快照已统一透出事件视图
5. 记忆刷新已具备消息增量 + 事件增量能力

- `premium` 回合

### 已完成的 P3 记忆增量化

- 会话状态已增加记忆游标能力：
  - `memoryCursor.lastMessageId`
