# 测试用例：事件管理

## 前置条件
- 已完成登录
- 已进入章节调试或游玩模式

## 测试步骤

### TC-004: 章节内容事件完整性验证
1. 进入章节调试模式
2. 观察第一个事件的显示
3. 检查事件属性：
   - index: 应该是 1
   - kind: scene
   - flow: chapter_content
   - status: 应该是 idle 或 active
   - summary: 显示事件描述

**预期结果：**
- 每个事件都有唯一的 index
- 事件状态流转清晰：idle -> active -> completed
- 事件有明确的开始-经过-结束过程

**验收标准：**
- [ ] 事件 index 唯一，不重复
- [ ] 事件状态正确流转
- [ ] 章节内容事件和结束条件事件分开显示
- [ ] 无故事件不会被标记为已完成

---

### TC-005: 结束条件事件验证
1. 进入章节调试模式
2. 检查结束条件事件的显示
3. 观察事件属性：
   - index: 应该是独立编号
   - kind: ending
   - flow: chapter_ending_check
   - status: active（等待判定）

**预期结果：**
- 结束条件事件有独立的 index
- 不会和章节内容事件混淆
- 状态显示为等待用户输入

**验收标准：**
- [ ] 结束条件事件 index 独立
- [ ] 结束条件事件 status 为 active
- [ ] 不会错误发送编排请求

---

### TC-006: 多章节事件隔离验证
1. 完成第一章节调试
2. 进入第二章节调试
3. 检查第二章节的事件列表

**预期结果：**
- 第二章节的事件从 index:1 开始
- 不会显示第一章节的已完成事件
- 事件状态都是初始状态

**验收标准：**
- [ ] 第二章节事件 index 从 1 开始
- [ ] 没有第一章节的残留事件状态
- [ ] 所有事件状态为初始状态

---

### TC-007: 事件状态流转验证
1. 进入章节调试模式
2. 观察第一个事件的状态
3. 用户输入响应后，观察事件状态变化
4. 事件完成后，观察下一个事件

**预期结果：**
- 事件状态按顺序流转：idle -> active -> completed
- 一个事件完成后，下一个事件自动激活
- 状态变化有明确的日志记录

**验收标准：**
- [ ] 事件状态正确流转
- [ ] 状态变化有日志记录
- [ ] 没有状态跳跃或混乱
