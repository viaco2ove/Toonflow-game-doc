## 二、分层架构详解

### 2.1 路由层 (routes/)

采用 **路由即控制器** 模式，每个文件导出一个 Express 路由处理器。

#### 路由自动发现机制

```
core.ts (路由自动发现)
    │
    ├── glob 扫描 routes/**/*.ts
    ├── 计算文件 hash 对比变化
    ├── 增量生成 router.ts
    └── 自动注册到 Express app
```

**优点：**
- 新增路由文件无需修改其他代码
- 增量编译，只重新生成变化的路由

#### 路由文件命名规范

```
routes/{模块}/{操作}{资源}.ts

例如：
├── routes/game/startSession.ts      # 开始会话
├── routes/game/saveWorld.ts         # 保存世界
├── routes/video/generateVideo.ts    # 生成视频
└── routes/novel/addNovel.ts         # 添加小说
```

### 2.2 服务层 (modules/game-runtime/services/)

负责有副作用的操作，主要是数据库读写和外部服务调用。

```
services/
├── SessionService.ts          # 会话核心服务
│   ├── 添加消息到会话
│   ├── 推进叙事回合
│   └── 管理会话生命周期
│
├── ChapterRuntimeService.ts   # 章节运行时服务
│   ├── 评估章节结局
│   ├── 计算章节进度
│   └── 管理章节状态
│
├── SnapshotService.ts         # 快照服务
│   ├── 创建状态快照
│   ├── 加载历史快照
│   └── 管理快照存储
│
└── SessionMemoryWorker.ts     # 后台记忆 Worker
    ├── 异步处理会话记忆
    ├── 定时执行优化
    └── 内存管理
```

### 2.3 引擎层 (modules/game-runtime/engines/)

无副作用的纯逻辑计算层，输入输出确定，不直接操作数据库。

```

│   ├── 协调 AI 对话流程
│   ├── 管理回合制叙事
│   └── 调度发言角色
│
├── ChapterProgressEngine.ts    # 章节进度引擎
│   ├── 计算章节完成度
│   ├── 状态转换逻辑
│   └── 进度评估
│
├── TriggerEngine.ts            # 触发器引擎
│   ├── 条件表达式求值
│   ├── 事件触发判定
│   └── 动作执行
│
├── TaskProgressEngine.ts       # 任务进度引擎
│   ├── 任务状态跟踪
│   ├── 成功/失败判定
│   └── 奖励发放
│
├── MiniGameController.ts       # 小游戏控制器
│   ├── 小游戏状态管理
│   └── 游戏规则判定
│
├── SpeakerRouteEngine.ts       # 发言路由引擎
│   ├── 确定发言角色
│   └── 角色切换逻辑
│
├── RuleOrchestrator.ts         # 规则编排器
│   ├── 规则组合执行
│   └── 策略调度
│
└── ChapterOutcomeEngine.ts     # 章节结局引擎
    ├── 结局条件判定
    └── 结局分支选择
```

## 四、请求处理流程

### 4.1 中间件链

```
Request
   │
   ▼
┌─────────────────────┐
│  logger (morgan)    │  ← 日志记录
└─────────┬───────────┘
          ▼
┌─────────────────────┐
│  cors               │  ← 跨域处理
└─────────┬───────────┘
          ▼
┌─────────────────────┐
│  express.json()     │  ← JSON 解析 (100MB limit)
│  express.urlencoded │  ← URL 编码解析
└─────────┬───────────┘
          ▼
┌─────────────────────┐
│  静态文件服务        │  ← uploads/ 目录
│  express.static     │
└─────────┬───────────┘
          ▼
┌─────────────────────┐
│  JWT 鉴权中间件      │  ← Token 验证
│  (白名单跳过)        │
└─────────┬───────────┘
          ▼
┌─────────────────────┐
│  资源隔离中间件      │  ← 用户数据隔离
└─────────┬───────────┘
          ▼
┌─────────────────────┐
│  路由处理器          │
└─────────┬───────────┘
          ▼
┌─────────────────────┐
│  404 处理           │
└─────────┬───────────┘
          ▼
┌─────────────────────┐
│  错误处理中间件      │
└─────────────────────┘
```

### 4.2 JWT 鉴权流程

```
1. 从 Authorization header 或 query.token 获取 token
2. 去除 "Bearer " 前缀
3. jwt.decode() 获取 userId
4. 查询 t_setting.tokenKey 获取签名密钥
5. jwt.verify() 验证签名
6. 将验证结果挂载到 req.user
7. 设置请求上下文 (requestContext)
```

### 4.3 资源隔离

```
每个请求携带 userId (来自 JWT)
所有数据库查询自动附加 userId 条件
防止用户间数据越权访问
```

## 五、数据库架构

### 5.1 ORM 选择

使用 **Knex.js** 作为查询构建器，配合 **better-sqlite3** 作为驱动。

```
Knex.js (查询构建器)
    │
    ▼
better-sqlite3 (同步驱动)
    │
    ▼
SQLite3 (嵌入式数据库)
```

### 5.2 数据库初始化

```
initDB.ts
    │
    ├── 创建连接
    ├── 执行建表 SQL
    ├── 插入种子数据
    │   ├── AI 模型配置
    │   ├── 提示词模板
    │   └── 语音预设
    └── 导出数据库实例
```

### 5.3 类型安全

```
database.d.ts (自动生成)
    │
    ├── 每个表对应一个 TypeScript interface
    ├── 字段类型映射
    └── 联合 DB 接口
         │
         ▼
    Knex 查询时获得类型提示
```

## 九、安全设计

### 9.1 数据隔离

- 用户通过 JWT 鉴权
- 所有查询附加 userId 条件
- resourceIsolation 中间件强制隔离

### 9.2 敏感信息保护

- API Key 存储在数据库，不硬编码
- .env 文件区分环境，不上传 git (local)
- tokenKey 每用户独立生成

### 9.3 输入校验

- 使用 Zod 进行请求体验证
- 文件大小限制 (100MB)
- 路径安全检查 (is-path-inside)
